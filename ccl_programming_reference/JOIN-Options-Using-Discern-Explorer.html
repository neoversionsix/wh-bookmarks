Join options provide an option to pull rows of data that generally do not qualify through the query join path. For example, when reading the ENCNTR_PRSNL_RELTN table to pull referring physicians, you might still want to see the patient's orders even if the patient does not have a referring physician on the table. At times, you might also want only those rows that do not have a matching row on the joining table. To accomplish either of these scenarios requires using a Join option. Below is a summary of the available Join options and when they are used. Links to method-specific documentation are provided for the appropriate syntax and related examples.

Standard Join

Standard Joins return the matches in the result set. A simple example is a query to get a list of people and their orders by joining the Person and Orders tables.

Methods

  • RDBMS/Non-RDBMS Tables
    • Link each table together on a field that contains a common value, A.FIELD = B.FIELD
  • RDBMS Tables
    • Nested Select allows the ability to create a SELECT command contained in another query that can be used to make additional qualifications. The subquery can be helpful when we want to qualify on tables but do not need to display any data from that table. This subquery is only available with RDBMS tables.
      • SUBQUERY: Evaluated once for the entire parent query
      • CORRELATED SUBQUERY: Evaluated once for each row processed
    • INNER JOIN in From Clause. See ANSI JOIN for more information.

OUTERJOIN

An OUTERJOIN returns the matches and exceptions in the result set. A simple example is a query to get a list of people that do and do NOT have orders. This example is accomplished by joining the Person table to the Orders table using the outer-join logic. Matches from both tables are returned and data that qualifies on the Person table is also returned, regardless of a match on the Orders table.

Exceptions

Exceptions return only the exceptions in the result set. A simple example is a query to get a list of people that do NOT have orders by Joining the Person and Orders tables, using exception logic to only qualify on people that do not have a row on the Orders table.

Methods

  • RDBMS Tables (Recommended Methods)
    • NOT EXISTS and Nested Select
    • OUTERJOIN function and Field = NULL
    • LEFT JOIN and Field = NULL
  • Non-RDBMS Tables (Not recommended when using RDBMS tables)
    • OUTERJOIN/DONTEXIST Control Option. See Discern Explorer OUTERJOIN for more information on using the OUTERJOIN control option.

Full (Outer) Join

The ANSI FULL JOIN returns rows that exist on both tables (standard join) or that exist on either table (LEFT JOIN or RIGHT JOIN). See ANSI JOIN for more information.

Methods

  • RDBMS Tables (Recommended Method)
    • FULL JOIN in From Clause

Explanation

Standard Join

Gathers only matches:

People that have orders.

  • RDBMS & Non-RDBMS Tables
    • A.FIELD = B.FIELD:
       PLAN P
       JOIN O WHERE O.PERSON_ID = P.PERSON_ID
  • Only RDBMS Tables
    • Nested Select:
       WHERE P.PERSON_ID = (
           SELECT O.PERSON_ID
           FROM ORDERS O
           WHERE O.CATALOG_CD = VAR1)
    • INNER JOIN in the From clause:
       FROM PERSON P,
           (INNER JOIN ORDERS O ON
               (O.PERSON_ID=P.PERSON_ID
               AND O.CATALOG_CD=VAR1))

OUTERJOIN

Gathers matches and exceptions:

People and if they have orders gather the orders.

  • Only RDMBS Tables
    • OUTERJOIN function
       PLAN P
       JOIN O WHERE O.PERSON_ID=OUTERJOIN(P.PERSON_ID)
    • LEFT JOIN in the From Clause
       FROM PERSON P,
           (LEFT JOIN ORDERS O ON (O.PERSON_ID = P.PERSON_ID
           AND O.CATALOG_CD IN (VAR1, VAR2)))
       PLAN P
       JOIN O
  • Non-RDBMS Tables
    • OUTERJOIN and DONTCARE Control Options
       FROM TABLEA A,
           NONRDBMS_TABLEB B
       PLAN A
       JOIN B WHERE A.FIELD = B.INDEXED_FIELD
       WITH OUTERJOIN = A

Exceptions

Gathers exceptions:

Only people that do NOT have orders.

  • Only RDBMS Tables
    • Not Exists and Nested Select
       PLAN P WHERE NOT EXISTS (
           SELECT O.PERSON_ID
           FROM ORDERS O
           WHERE O.PERSON_ID = P.PERSON_ID)
    • OUTERJOIN and Field = NULL
       PLAN P
       JOIN O WHERE O.PERSON_ID=OUTERJOIN(P.PERSON_ID)
           AND O.PERSON_ID = NULL
    • LEFT JOIN and Field = NULL
       FROM PERSON P,
           (LEFT JOIN ORDERS O ON (
               PERSON_ID = P.PERSON_ID AND
               O.CATALOG_CD IN (VAR1, VAR2)))
       PLAN P
       JOIN O WHERE O.PERSON_ID = NULL
  • Non-RDBMS Tables
    • OUTERJOIN and DONTEXIST Control Options
       FROM TABLEA A,
           TABLEB B,
           NONRDBMS_TABLEC C
       PLAN A
       JOIN B WHERE A.FIELD = B.INDEXED_FIELD
       JOIN C WHERE A.FIELD = C.INDEXED_FIELD
       WITH OUTERJOIN = B, DONTEXIST

Full (Outer) Join

Gathers matches and exceptions from tables:

All items on the EXPLORER_MENU table, all items on the CCL_CUST_SCRIPT_OBJECTS table, along with the matches between those tables, which would be the custom programs that also have been added to Explorer Menu.

  • Only RDBMS Tables
    • FULL JOIN in the From Clause
       SELECT MENU_ID = E.MENU_ID,
           MENU_OBJECT = E.ITEM_NAME,
           MENU_TYPE = E.ITEM_TYPE,
           CUST_NAME = C.OBJECT_NAME,
           CUST_ID = C.CCL_CUST_SCRIPT_OBJECTS_ID
       FROM CCL_CUST_SCRIPT_OBJECTS C,
           (FULL JOIN EXPLORER_MENU E ON TRIM(E.ITEM_NAME) = C.OBJECT_NAME)
       ORDER E.ITEM_NAME, C.OBJECT_NAME

Note

Typically, you do not not want to wrap a function around an indexed field in a linking statement as it removes the field as a choice from the optimizer. This example uses it because the ITEM_NAME field is a C30 and the OBJECT_NAME is a VC30. The TRIM function allows the C30 field to appropriately link to the VC30 field which is also indexed.
