Two run-time parameters are defined in the SELECT clause of the sample program, as shown below:

SELECT INTO $1
	P.NAME_LAST_KEY
FROM	PERSON P
WHERE	P.NAME_LAST_KEY= PATSTRING(CNVTUPPER($2))

Note

The above example is a syntax example. To use run-time parameters, you must create an executable program.

The first prompt accepts an output device and assigns the value to the $1 parameter. The second prompt accepts a last name and assigns the value to the $2 parameter. With these parameters, you can specify where the output is directed and what last names qualify. Notice that the CNVTUPPER() and PATSTRING() functions are used on the $2 parameter. Since character data is case sensitive, "SMITH" does not equal "Smith". You can use the CNVTUPPER( ) function on the parameter to convert the user input to uppercase. Suppose that the user wanted to select all names beginning with "SMITH", so the user entered "smith*" at the prompt. The CNVTUPPER( ) function converts the input to uppercase but now does not recognize the "*" as a wildcard. The PATSTRING( ) function allows using the "*" as a wildcard to perform pattern string matches on the data that is converted to uppercase. So entering "smith*" qualifies last names beginning with "SMITH".

If you are not using prompts in your program, you must specify the run-time parameters following the name of the program when you execute the program from the command line. For example, if we took the prompts out of the above example program, we could use the following command to execute the program and extract people whose last name is "Jones" and save the output in a file.

PAT_NAMES "MY_FILE", "JONES" GO

The first parameter "MY_FILE" is used in place of $1 in the program. Selecting into "MY_FILE" directs the output to a file in the default directory named "MY_FILE.DAT". The second parameter "JONES" is used in place of $2 in the program. Only rows where the NAME_LAST_KEY column is equal to "JONES" qualifies.

When entering in a string at a prompt, it is not necessary to put quotes around the string. The reason for this is because Discern Explorer looks at the default value for the prompt and assumes that you are entering in a value of the same data type. So in the above example, entering in "smith*" or smith* at the second prompt returns the same results.
