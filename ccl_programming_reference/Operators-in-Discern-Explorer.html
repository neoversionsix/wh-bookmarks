IN

Multiple OR qualification

Enables you to specify a value in one or more expressions. P.NAME_LAST_KEY IN ("SMITH", "ROSE", "TATE").  

By default, an IN clause that uses three or fewer items and pattern-matching wildcard characters is converted to a series of OR clauses using the LIKE operator. If an IN clause is not converted and remains as is, pattern matching is not allowed because the Relational Database Management System (RDBMS) requires the LIKE clause, and LIKE cannot be used with an IN clause. In theory, you should not use pattern-matching strings in an IN clause, even though it is allowed for three or less items. Beginning in Discern Explorer versions 8.17.5 and 9.05.5 a warning is generated if an IN clause containing wildcard pattern-matching characters is not converted to OR clauses using the LIKE operator. SET TRACE INOPTIMIZEOR int is used to override the default number or items in an IN clause that are converted to OR clauses.

LIKE

Pattern Match

P.NAME_LAST_KEY LIKE "SM*"
When qualifying on a field from an RDBMS table, you can use the equal operator (=) and a wild card character for pattern matching. For example, P.NAME_LAST_KEY = "SM*" will have the same result as P.NAME_LAST_KEY LIKE "SM*".

To calculate a square root, raise a value to the power of 0.5; for example, to calculate the square root of 25, use 25**0.5.


Note: When using the division operator in a Discern Explorer program, avoid dividing by a denominator of zero. While the mathematical expression results in an undefined value when dividing by zero, Discern Explorer automatically checks for the error condition and assigns a value of zero if a denominator of zero is applied in a calculation. This indicates a previous logical failure in the script when this condition occurs. The result of a division by zero is undefined, and programs should validate the input appropriately when performing calculations. Beginning in Discern Explorer versions 8.15.4 and 9.03.4 a warning message is generated when a divide by zero is attempted.

Operator

Operation

Arithmetic Left Operators are introduced in version 8.11.3 of Discern Explorer and can be used to change the value of a variable, an array, or record structure item. The initial functionality does not support using the Arithmetic Left Operators to change the value of an array or record structure item in a Report Writer clause. Discern Explorer version 8.12.2 extends the functionality to support using the Arithmetic Left Operators to change the value of an array or record structure item in a Report Writer clause. Discern Explorer versions 8.14.1 and 9.02.1 extend the functionality to support using namespaces in a Report Writer clause. Before Discern Explorer versions 8.15.4 and 9.03.4 using the Arithmetic Left Operators to set a variable that is not previously declared does not set the variable but no warning or error is generated. Beginning with Discern Explorer versions 8.15.4 and 9.03.4 using the Arithmetic Left Operators to set a variable that is not previously declared generates a report variable has not been initialized error.

Discern Explorer versions 8.14.3 and 9.02.3 extend the functionality to support the Arithmetic Left Operators for W8 data types in 64-bit AIX environments.


When a division by zero is attempted in a Discern Explorer program, using left operator syntax or compiled with optimized syntax, Discern Explorer versions 9.03.3, 8.15.3 and later check for the error condition and assign a value of zero. Discern Explorer 9.03.2, 8.15.2 and prior versions assign the operating system return value of NaN, -nan, or inf. The result of a division by zero is undefined, and programs should validate the input appropriately when performing calculations. Beginning in Discern Explorer versions 8.15.4 and 9.03.4 a warning message is generated when a divide by zero is attempted.

Parentheses can be used to override the default order of precedence. For example, 1 + 2 * 3 evaluates to 7 because the precedence of multiply (*) is before the precedence of add (+). The expression 1 + 2 * 3 is evaluated by multiplying 2 and 3, and then adding 1 for a result of 7. Adding parentheses causes the expression of (1 +2) * 3 to evaluate to 9. In this example, 1 is added to 2 and then multiplied by 3, so the final result evaluates to 9.
