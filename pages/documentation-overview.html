Cerner Millennium Core

Core is comprised of common processes or tasks shared multiple times across clinical domains, technical activities, and project tasks.

Focus: "Leverage through reuse" — e.g., Patient Care Hierarchy built once and used in multiple solutions.

Core Elements

  • Organizations and Locations — Patient Care Hierarchy, Organization Types
  • Interface Processing — Contributor Systems, Contributor Sources, Code Sets, Code Values, Aliases
  • Order Catalog — Descriptions, Synonyms
  • Results Processing — Event Set Hierarchy
  • Security — Workflow, Security Types

Client/Server Model

Cerner Millennium utilizes a Client/Server Model — distribution of computing tasks, cooperative computing between clients requesting information and servers "serving" information.

This is a three-tiered, distributed, client-server model.

Tier 1 — Client

  • Also known as: Front-end, GUI
  • Windows operating system
  • End-user solutions/tools accessible: PowerChart Surgery Management Core Code Builder Bedrock Access HIM

Tier 2 — Middleware

  • Also known as: Main Frame Server, Node, Host, Back-end
  • UNIX (HP-UX, AIX, or Linux) operating system
  • Software programs performing multiple functions including connectivity and data transfer
  • Programs running on back-end are "processes"; a subset of these are "application servers" providing services including data transfer and connectivity between Tier 1 and Tier 3

Tier 3 — Database

  • Managed by Oracle relational database management system (RDBMS)
Example: A user places an order through Tier 1, the request passes through Middleware to the Database, and the Database accepts and updates.
Three-Tier Client/Server Model

Database Data Types

Three types of data exist in the database:

1. Reference Data

Information built once and used repeatedly. Examples: Patient Types, Languages, States, Patient Locations.

2. Activity Data

Patient-specific information. Examples: Orders and Results.

3. Metadata

Information about the database itself. Examples: Table space, Table structure.

Common Data Model (CDM)

The Common Data Model is a map of tables in the database. It defines the relationship of data within the system.

Data Flow

  1. End user makes a request at the Client tier
  2. Request is communicated to the Database through the Middleware
  3. Once information is retrieved from the Database, it is sent back through the Middleware to the Client
Example — Patient Search in PowerChart: Middleware takes the request and passes it to the Database. The Database accepts and gathers the requested information. Middleware receives the reply and passes it to the Client. The Client displays all patients for the end user.
Data Flow: Request and Response

Database Tables

  • Cerner Millennium has over 5800 tables
  • Tables are comprised of rows and columns
  • Goal is to store data values once
  • Columns are also known as "attributes"
  • Rows are also known as "records"
  • Data values are found at the intersection of a row and column

Primary and Foreign Keys

Primary Keys

  • Always unique, system-generated, numeric identifiers
  • First column on a table
  • Usually the table name followed by _id
  • PERSON's PK is person_id; PERSON_ALIAS's PK is person_alias_id

Foreign Keys

  • Primary keys from other tables
  • Used to link tables together
  • Many tables have similar foreign keys
  • person_id is a FK on both PERSON_ALIAS and ENCOUNTER tables
Relationship: One-to-Many relationship between PK and FK. Also called a "Parent and Child" table relationship.

Discern Explorer (CCL)

Also known as CCL (Cerner Command Language). A language patterned after SQL used to communicate with the database.

Used To

  • Perform database queries
  • Troubleshoot issues
  • Design reports
  • Build programs

Accessible Via

  • Discern Visual Developer
  • Back-end

Basic Query Structure

Select * from <table> where <attribute> = data value
ComponentDescription
SelectUse "select" to start every query
<attribute> or *List attribute name or use * as wildcard
from <table>Use "from" and the table name
whereQualifying section, limits returns
<attribute>Attribute corresponding to data value
= data valueUse = and list the known data value

Example Queries

Select * from PERSON where name_last_key = "YOUNG"

Select * from PERSON_ALIAS where alias = "7082"

Select alias from ENCNTR_ALIAS where alias = "645017032"

Select * from PERSON where person_id = 12856100

Select * from CODE_VALUE where code_value = 309311

Select alias from ENCNTR_ALIAS where encntr_id = 97725398
Rules for data value formats: Any attribute that has the potential to store text must have quotes around its data value. This includes _KEY attributes and ALIAS attributes.

Steps to Understanding Queries

  1. What data do you have and what table is it located within?
  2. What data do you want and what table is it located within?
  3. What is the connection between these tables? (direct or indirect relationship)
Indirect Relationship Example: PERSON and ENCNTR_ALIAS are related through the ENCOUNTER table (an indirect table relationship).
