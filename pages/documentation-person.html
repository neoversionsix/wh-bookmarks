Person Information – How It Gets into the Database

Person information can originate from a Cerner Millennium Solution or a Contributor System. Both write to the Cerner Millennium Database.

  • Person information is written to the PERSON table and PERSON_ALIAS table
  • One active row per person on the PERSON table (main goal of the database)
  • Additionally consolidating other person information
Person Information Flow

Person Alias Pools

Person aliases are used for identification:

  • MRN – Medical Record Number
  • SSN – Social Security Number
  • SIN – Social Insurance Number

These aliases uniquely identify a person.

Alias Pools are used when an alias cannot be pre-defined or anticipated. Much of received patient information from an interface is predefined (reference data), such as Encounter type, Language Spoken, and County.

Aliases and Alias Pools Analogy

Alias pools are like "decks of cards" within the system. Each alias is a "card from a particular deck."

Alias Pool Types

Five core alias pool types:

  1. Person – MRN, CMRN, SSN
  2. Encounter – FIN
  3. Personnel – Personnel ID, Doctor ID, NPI, DEA
  4. Order – Order ID
  5. Organization – Encounter Org Alias Pool
Additional alias pool types exist for solution-specific functionality.

Purpose of Alias Pools

Stop Duplication in the Cerner Millennium Database.

Uniquely identify whether the system has seen a particular person, personnel, encounter, order, or organization alias before.

Processing Patient Transactions

Cerner Millennium handles vast amounts of patient information transactions every day. With each transaction, the system attempts to consolidate the database to run efficiently.

Main Goal: One active row per person on the PERSON table.

Person identification is a two-step process:

  1. Match – Required for every interface sending data
  2. Reconciliation – Optional, occurs only when match is not successful

Person Match and Reconciliation

Automated processes to combine person information within the database. Can also occur manually via HNACombine.exe.

Transaction Processing Flow

Primary Identifiers

Primary identifiers establish the identity of a patient or encounter.

Criteria

  • Exist for all patients
  • Cannot have null values
  • Must be unique

Levels

  • Person-Level: MRN most commonly used
  • Encounter-Level: FIN most commonly used
The combination of Alias + Alias Pool makes a person or encounter unique.

Match with Primary Identifiers

  • Person match: MRN (Alias) + where MRN came from (Alias Pool) = unique person
  • Encounter match: FIN (Alias) + where FIN came from (Alias Pool) = unique encounter

Match Process

The database searches for existing records with matching primary identifiers.

  • Match found (YES): Information added/updated to existing record
  • No match (NO): New records added to appropriate tables

Person Transaction Example

Same person John Smith visiting TWMC (MRN 123456 from TWMC MRN pool) and Stagecoach (MRN 123456 from Stagecoach MRN pool).

Initially treated as two separate people because the combination of Alias + Alias Pool makes each unique. MRN from TWMC is not the same as MRN from Stagecoach.

Reconciliation

Reconciliation is needed when match is unsuccessful. It uses Secondary Identifiers.

Interface Authentication

  • Authenticated interfaces: Trusted/originating source of demographic data (usually registration system)
  • Unauthenticated interfaces: Not a trusted source (Lab, Radiology systems)

Secondary Identifiers

  • First Name
  • Last Name
  • Date of Birth
  • Sex
  • SSN
Secondary identifiers may not be unique and may be missing or null. Clients determine which secondary identifiers to use.

Resolution

If all designated secondary identifiers match AND both interfaces are authenticated, the system combines the two records into one.

Example: John Smith, DOB 01-11-1983, Male, SSN 123-45-6789

Matching Revisited – Two Steps

Step 1 – Person Match (Primary Identifier: MRN + Alias Pool): "Has the database seen this person before?"

  • YES: Proceed to Encounter Match (Primary Identifier: FIN + Alias Pool) – "Has the database seen this encounter before?"
  • NO: Proceed to Reconciliation
Two-Step Matching Process

Results of Person/Encounter Matching

  • Person match unsuccessful: New records written to PERSON, PERSON_ALIAS, ENCOUNTER, ENCNTR_ALIAS
  • Encounter match unsuccessful: New records only to ENCOUNTER, ENCNTR_ALIAS (referencing matched person)
  • Both successful: Database updates existing Person/Encounter rows

Reconciliation Revisited

If person match is unsuccessful, the system attempts reconciliation using secondary identifiers IF:

  • Information is from an "authenticated" feed
  • ALL secondary identifiers match (across entire database, all alias pools)

Consolidation After Successful Reconcile

  • Old person's person_id is inactivated
  • Replace every reference to old person_id with new person_id (except PERSON table)
  • Duplicate SSNs on PERSON_ALIAS are inactivated (old person's SSN)

Directional Combine Logic

Provides key features for combining records:

  • Automatically determines most efficient process to combine person/personnel records
  • Retains selected demographic data
  • Always maintains personnel row when included

Three Options

  1. HNACombine.exe
  2. Interface (A34) combines
  3. Reconciliation combines

Engaged when taking 2015.01.19 code and above.

Wrap Up

  • The system can un-combine records
  • No rows are deleted, only inactivated
  • The system cross-references old vs new rows
  • Clients designate Aliases and Alias Pools used
  • Goal: One active row per person on the PERSON table
