The Electronic Medical Record

The Cerner Millennium database stores information for patients from the EMR, including Visits, Orders, Results, and more.

Positives

  • Represents continuum of care across facilities
  • Increases quality of patient care
  • Efficient access of patient information by anyone treating the patient on site

EMR Risks

Unauthorized Access to Patient Information

  • Patient demographic information
  • Sensitive results: Mental Evaluations, Pregnancy Tests, Toxicology Reports

Unauthorized Disclosure of Organization Information

  • Organization information or trade secrets
  • Drug studies for ground-breaking treatments

Data Corruption

  • Software virus
  • Natural disasters (floods, hurricanes, fires)
The Reality: No method of security is 100% secure. There are many different and robust methods to mitigate risks in Cerner Millennium.

Security Workflow and Types

When a user makes a request for data through an application, it triggers the security workflow:

Security Workflow

Step 1: Authentication Security

  • Identifies the user
  • Username, Password, Domain Access
Key Question: "Are you who you say you are?"

Step 2: Application Authorization

Also known as Application Task Security. Occurs after authentication and determines which applications and tasks a user can access based on Positions and Applications.

Positions

  • Built for groups of users with similar access needs
  • Examples: Physician, Hospitalist, Medical Technician, Unit Secretary
  • Each user will have one position
  • Building Tools: Core Code Builder (Code Set 88), Bedrock Position Wizard

Application Groups

  • Groups of tasks associated to a position
  • Individual tasks in an application may be granted or revoked
  • One or more application groups associated to each position
  • All users with the same position have the same application access
  • Tool: Task Access tool

Step 3: Encounter Access Security

Two security types (both are domain-wide settings, not required):

Type 1: Encounter/Organization Security

  • Allows user to see encounters and data from assigned organizations only
  • Each user can be associated to one or multiple organizations
  • Domain-wide setting
Example: A user associated to TWMC can see TWMC encounters. If a patient had encounters at both TWMC and Stagecoach, the user only sees the TWMC ones.

Type 2: Confidentiality Level Security

  • Encounter/Organization Security MUST be utilized first
  • Enhances encounter/org security by offering more selective security around encounters
  • Compares confidentiality level of user per organization against the encounter
  • User must be same level or higher to access the encounter
Use Cases: Child abuse cases, Psychiatric visits, Crime victims, Legal issues, Celebrities

Confidentiality Level Example

Organization User Level Encounter Level Access?
TWMC 2 3 DENIED (2 < 3)
Stagecoach 4 3 GRANTED (4 ≥ 3)
Longbranch 3 3 GRANTED (3 ≥ 3)

Step 4: Chart Access Security

Key Concept: "It's all about Relationships"
  • A relationship is required to open a patient chart
  • Relationship defines affiliation between clinician and patient/encounter
  • Relationships are associated to a position
  • One-to-One or One-to-Many

Two Types of Relationships

Lifetime (Person Level)

Code Set 331

Expirable (Encounter Level)

Code Set 333

Emergency Access

  • Users may be allowed to identify if additional encounters exist
  • User with Emergency Access: can see hidden encounters
  • User without Emergency Access: cannot see hidden encounters

Emergency Access Override Configuration

  • Used to override encounter/organization and confidentiality-level securities
  • If active, there are other encounters currently hidden
Requirements:
  • Emergency Access Relationship (Code Set 331)
  • Privilege to self-declare
  • Emergency Access Relationship associated to user's position

Step 5: Data Access

Key Concept: "It's all about Privileges"
  • Refinement of Application/Task Access (Step 2)
  • Allows modified access to application/task
  • Determines what specific data is viewable in patient chart
  • Determines which functions are performed by end-user within PowerChart framework solutions
  • Build Tool: PrivMaint.exe

Examples of Privileges in Data Access

  • Application/task access determines that I can place orders, but privileges determine which orders I can place
  • Application/task access determines that I can view all results, but privileges determine which results I can view

Location Security

Additional Strategy: Location Security

  • Devices associated with a location within Millennium
  • Location provides location-specific functionality with certain solutions
  • Defines default printers
  • Does NOT restrict users from viewing patients in other locations

Front-End Tools for Building Users

Tool Type Purpose
HNAUser Tool Front-end tool Allows single users to be built. Ideal for maintenance.
Bedrock Personnel Wizard Front-end wizard Adding and maintaining users. Batch update of personnel associations to facilities, specialties, and positions.
Content Manager Front-end tool (spreadsheet) Allows mass uploads, unloads, and multi-user additions and modifications.

Accountability Strategies

Designed to ensure timely and responsible access to patient data.

Auditing Reports (Back-end)

  • Self-declared relationships
  • Charts accessed by a user
  • Users accessing a particular patient chart
  • Report on users with no activity
  • Customized reports using Discern Explorer
